## state란 무엇인가

state는 **화면에 영향을 주는 값**을 의미한다.
일반 변수는 값을 바꿔도 화면이 자동으로 바뀌지 않지만, state는 변경되면 React가 다시 렌더링을 수행한다.

```jsx
const [count, setCount] = useState(0);
```

위 코드에서 `count`는 현재 상태값이고, `setCount`는 상태를 변경하는 함수이다.
React에서는 화면에 반영되어야 하는 값이라면 일반 변수보다 state로 관리하는 것이 기본이다.

즉, React에서는 **값을 바꾸는 것 자체보다, 그 값이 화면에 영향을 주는 값인지**를 먼저 구분해야 한다.

## 렌더링이란 무엇인가

렌더링은 컴포넌트 함수를 다시 실행하여 화면에 보여줄 결과를 다시 계산하는 과정이다.

예를 들어 아래와 같은 컴포넌트가 있다고 가정한다.

```jsx
function Counter() {
  const [count, setCount] = useState(0);

  return <button onClick={() => setCount(count + 1)}>{count}</button>;
}
```

버튼을 클릭하면 `setCount`가 호출된다.
그러면 React는 `count`가 바뀌었다고 판단하고 `Counter` 컴포넌트를 다시 실행한다.
그 결과 새로운 JSX가 계산되고, 바뀐 부분만 실제 화면에 반영된다.

즉 React의 기본 흐름은 다음과 같다.

* state 변경
* 컴포넌트 재실행
* JSX 재계산
* 변경된 화면 반영

## 부모와 자식의 관계

React에서는 보통 **state를 부모가 가지고**, 자식은 그 값을 props로 전달받는다.

```jsx
function Parent() {
  const [isDone, setIsDone] = useState(false);

  return <Child isDone={isDone} />;
}
```

```jsx
function Child({ isDone }) {
  return <input type="checkbox" checked={isDone} />;
}
```

이 구조에서 `isDone`의 실제 소유자는 부모이다.
자식은 부모가 전달해준 값을 받아서 화면에 보여주기만 한다.

즉 props는 자식이 스스로 만드는 값이 아니라, **부모가 내려주는 값**이다.

## 자식이 부모의 값을 바꾸는 방법

자식은 부모의 state를 직접 바꿀 수 없다.
대신 부모가 state를 변경하는 함수를 만들고, 그 함수를 자식에게 props로 내려준다.

```jsx
function Parent() {
  const [isDone, setIsDone] = useState(false);

  const onUpdate = () => {
    setIsDone((prev) => !prev);
  };

  return <Child isDone={isDone} onUpdate={onUpdate} />;
}
```

```jsx
function Child({ isDone, onUpdate }) {
  return <input type="checkbox" checked={isDone} onChange={onUpdate} />;
}
```

이때 흐름은 다음과 같다.

* 부모가 state를 가지고 있음
* 자식이 이벤트를 발생시킴
* 자식이 부모로부터 받은 함수를 호출함
* 부모의 state가 변경됨
* 부모가 다시 렌더링됨
* 자식도 새로운 props를 받아 다시 렌더링됨

즉 React의 데이터 흐름은 기본적으로 **부모에서 자식으로 내려가는 단방향 흐름**이다.

## 체크박스의 checked와 제어 컴포넌트

체크박스에 아래와 같이 `checked`를 지정하면, 이 체크박스는 React state에 의해 제어된다.

```jsx
<input type="checkbox" checked={isDone} onChange={onUpdate} />
```

이 경우 체크 여부는 브라우저가 독립적으로 결정하지 않고, `isDone` 값이 결정한다.
즉 클릭했다고 바로 체크 상태가 확정되는 것이 아니라, 다음 흐름으로 동작한다.

* 사용자가 체크박스를 클릭함
* `onChange`가 실행됨
* state가 변경됨
* 컴포넌트가 다시 렌더링됨
* `checked={isDone}` 결과가 화면에 반영됨

따라서 state가 바뀌지 않으면 체크박스도 원하는 대로 동작하지 않는다.
이런 입력 요소를 **제어 컴포넌트**라고 이해하면 된다.

## 배열과 객체 state는 직접 수정하지 않아야 하는 이유

React에서는 state가 배열이나 객체일 때 원본을 직접 수정하지 않는 것이 중요하다.
기존 값을 직접 바꾸면 React가 변경을 제대로 감지하지 못하거나, 예측하기 어려운 동작이 생길 수 있다.

좋지 않은 방식은 다음과 같다.

```jsx
todos[0].isDone = true;
setTodos(todos);
```

이 방식은 기존 배열을 직접 수정한 것이다.

권장되는 방식은 새로운 배열과 새로운 객체를 만들어서 state를 변경하는 것이다.

```jsx
setTodos(
  todos.map((todo) =>
    todo.id === targetId ? { ...todo, isDone: !todo.isDone } : todo
  )
);
```

이 방식은 기존 배열을 그대로 두고, 변경이 필요한 요소만 새 객체로 만들어 반환한다.
React에서는 이런 식의 **불변성 유지 방식**이 기본이다.

## 리스트 렌더링에서 key가 중요한 이유

배열을 화면에 출력할 때는 각 요소를 구분할 수 있는 `key`가 필요하다.

```jsx
{todos.map((todo) => (
  <TodoItem key={todo.id} {...todo} />
))}
```

`key`는 React가 각 항목을 식별하기 위한 기준이다.
이 값이 불안정하거나 중복되면 체크박스 상태, 입력값, 렌더링 결과가 이상하게 보일 수 있다.

따라서 리스트를 렌더링할 때는 index보다 **고유한 id**를 key로 사용하는 것이 좋다.

## Reducer란 무엇인가

Reducer는 **상태를 어떻게 바꿀지 정해놓은 함수**이다.
이전 state와 어떤 요청(action)을 받아서, **새로운 state를 반환**하는 역할을 한다.

```jsx
import { useReducer } from "react";

function reducer(state, action) {
  switch (action.type) {
    case "INCREASE":
      return state + 1;
    case "DECREASE":
      return state - 1;
    case "RESET":
      return 0;
    default:
      return state;
  }
}

export default function Counter() {
  const [count, dispatch] = useReducer(reducer, 0);

  return (
    <div>
      <h1>{count}</h1>
      <button onClick={() => dispatch({ type: "INCREASE"})}>증가</button>
      <button onClick={() => dispatch({ type: "DECREASE" })}>감소</button>
      <button onClick={() => dispatch({ type: "RESET" })}>초기화</button>
    </div>
  );
}
```

여기서 중요한 점은 버튼이 직접 state를 바꾸지 않는다는 것이다.

버튼은 그냥 이렇게 요청만 보낸다.

```jsx id="nk67pd"
dispatch({ type: "INCREASE" })
```

그러면 실제 상태 변경은 reducer가 담당한다.

reducer은 상태 변경 규칙을 모아두는 역할이다.

---

## reducer를 이해할 때 꼭 알아야 하는 점

### reducer는 새로운 state를 반환해야 한다

이전 state를 직접 수정하면 안 된다.

안 좋은 예:

```jsx
function reducer(state, action) {
  state.count += 1;
  return state;
}
```

좋은 예:

```jsx
function reducer(state, action) {
  return {
    ...state,
    count: state.count + 1,
  };
}
```

즉 reducer도 state를 직접 바꾸는 게 아니라
**새로운 state를 만들어 반환**해야 한다.

# Context란 무엇인가

Context는 **부모가 가진 값을 중간 컴포넌트들을 거치지 않고, 아래쪽 여러 자식 컴포넌트들이 직접 꺼내 쓸 수 있게 해주는 기능**이다.

React는 기본적으로 부모가 자식에게 `props`를 통해 값을 전달한다.
그런데 컴포넌트 구조가 깊어지면, 실제로 쓰지도 않는 중간 컴포넌트들이 계속 props를 전달만 하게 되는 상황이 생긴다.

이런 문제를 해결하기 위해 Context를 사용한다.

즉 Context는 다음 상황에서 등장한다.

* 어떤 값을 여러 컴포넌트가 함께 써야 함
* 컴포넌트 깊이가 깊어서 props 전달이 번거로움
* 중간 컴포넌트는 그 값을 쓰지 않는데 계속 전달만 해야 함

이런 구조를 보통 **props drilling**이라고 부른다.

---

# Context 생성

```jsx
import { createContext } from "react";

export const UserContext = createContext();
```

# Provider로 값 공급

```jsx
import { UserContext } from "./UserContext";

function App() {
  const user = { name: "Kim" };

  return (
    <UserContext.Provider value={user}>
      <Layout />
    </UserContext.Provider>
  );
}
```

이 Provider 아래에 있는 컴포넌트들은 `user` 값을 꺼내 쓸 수 있다.

즉 Provider는

> 이 아래 자식들에게 이 값을 공유하겠다

라는 의미이다.

# 자식에서 사용

아래쪽 컴포넌트에서는 `useContext`로 값을 꺼낸다.

```jsx id="n6st1u"
import { useContext } from "react";
import { UserContext } from "./UserContext";

function Profile() {
  const user = useContext(UserContext);

  return <div>{user.name}</div>;
}
```

# 어떤 값을 Context에 넣는가

보통 이런 값들을 Context로 많이 관리한다.

* 로그인한 사용자 정보
* 다크 모드 여부
* 언어 설정
* 전역 알림 상태
* Todo 관련 공통 함수나 데이터

즉 앱 전체나 특정 범위에서 **여러 컴포넌트가 함께 써야 하는 값**이 대상이다.

반대로 아주 한정된 곳에서만 쓰는 값은 굳이 Context로 뺄 필요가 없다.

예를 들면 이런 것은 보통 그냥 `useState`와 props로 충분하다.

* 특정 버튼의 열림/닫힘
* 한 input의 입력값
* 특정 컴포넌트 내부에서만 쓰는 상태

# Context도 state를 바꾸는 데 사용할 수 있는가

가능하다.
실제로는 값만 넣는 것이 아니라, **값과 상태 변경 함수**를 함께 넣는 경우가 많다.

예를 들면 다음과 같다.

```jsx id="d5qsk2"
import { createContext, useState } from "react";

export const TodoContext = createContext();

function App() {
  const [todos, setTodos] = useState([]);

  const onUpdate = (targetId) => {
    setTodos((prevTodos) =>
      prevTodos.map((todo) =>
        todo.id === targetId ? { ...todo, isDone: !todo.isDone } : todo
      )
    );
  };

  return (
    <TodoContext.Provider value={{ todos, onUpdate }}>
      <TodoList />
    </TodoContext.Provider>
  );
}
```

그리고 아래쪽에서는 이렇게 쓴다.

```jsx id="tmu58s"
import { useContext } from "react";
import { TodoContext } from "./TodoContext";

function TodoItem({ id, isDone, content }) {
  const { onUpdate } = useContext(TodoContext);

  return (
    <div>
      <input
        type="checkbox"
        checked={isDone}
        onChange={() => onUpdate(id)}
      />
      <span>{content}</span>
    </div>
  );
}
```

즉 Context는 데이터를 담을 수도 있고,
함수를 담아서 아래 컴포넌트가 호출하게 만들 수도 있다.

# Context와 useReducer를 같이 쓰는 이유

React에서 자주 나오는 조합이 **Context + useReducer**이다.

이 조합을 쓰는 이유는 역할이 다르기 때문이다.

* `useReducer`: 상태를 어떻게 바꿀지 관리함
* `Context`: 그 state와 dispatch를 여러 컴포넌트에 전달함

예를 들면 이런 구조이다.

```jsx 
const TodoContext = createContext();
```

```jsx
function App() {
  const [todos, dispatch] = useReducer(reducer, initialState);

  return (
    <TodoContext.Provider value={{ todos, dispatch }}>
      <TodoEditor />
      <TodoList />
    </TodoContext.Provider>
  );
}
```

아래 컴포넌트는 이렇게 쓸 수 있다.

```jsx id="jlwm6r"
const { todos, dispatch } = useContext(TodoContext);
```

즉 Context는 전달 역할, Reducer는 상태 변경 규칙 역할을 담당한다.

---

# Context를 쓴다고 완전한 전역 상태 관리가 되는가

작은 규모에서는 거의 그렇게 느껴질 수 있다.
하지만 Context는 어디까지나 **값을 전달하는 도구**이다.

즉 Context 자체가 상태를 저장하는 것은 아니고,
실제로 상태는 여전히 `useState`나 `useReducer`가 관리한다.

정리하면 다음과 같다.

* Context: 전달 담당
* useState/useReducer: 상태 관리 담당

이 둘을 같이 써야 의미가 커진다.

---

# 주의할 점

## 모든 값을 Context로 넣을 필요는 없다

처음 배우면 공통값은 다 Context로 빼고 싶어질 수 있다.
그런데 너무 많이 넣으면 오히려 구조가 복잡해진다.

기준은 이렇다.

* 여러 컴포넌트가 자주 함께 쓰는가
* 깊게 전달해야 해서 props가 번거로운가

이럴 때만 Context를 고려하면 된다.

---

## Provider 바깥에서는 useContext 값이 없다

어떤 컴포넌트가 Context 값을 쓰려면 반드시 해당 Provider 내부에 있어야 한다.

예를 들어 Provider 밖에서 `useContext(UserContext)`를 호출하면 원하는 값이 안 나온다.

즉 Context는 앱 전체 어디서나 자동으로 보이는 것이 아니라,
**Provider가 감싸고 있는 범위 안에서만 유효**하다.

---

## 값이 바뀌면 관련 컴포넌트들이 다시 렌더링된다

Provider의 `value`가 바뀌면, 그 Context를 사용하는 컴포넌트들이 다시 렌더링된다.
그래서 너무 많은 값을 한 Context에 몰아넣으면 렌더링 관리가 어려워질 수 있다.

처음엔 깊게 신경 쓰지 않아도 되지만, 나중에는

* Context를 나누거나
* value를 메모이제이션하거나
* 상태를 분리하는 방식

같은 최적화도 하게 된다.
