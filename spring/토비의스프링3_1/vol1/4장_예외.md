# 4장. 예외
## 1. 예외의 종류와 특징
### 1.1. Error
* `java.lang.Error` 클래스의 서브 클래스들
* 시스템에서 비정상적인 상황이 발생했을 때 사용됨
* 애플리케이션 코드로 잡으려고 하면 안됨

### 1.2. Exception
* `java.lang.Exception` 클래스의 서브 클래스들
* 애플리케이션 코드에서 잡아서 처리할 수 있는 예외들

#### 1) RuntimeException(Unchecked Exception)
* `java.lang.RuntimeException` 클래스의 서브 클래스들
* 프로그램에서 논리적인 오류가 있을 때 발생하도록 의도된 예외
* 애플리케이션 코드에서 잡아서 처리할 수 있지만, 잡지 않아도 되는 예외들

#### 2) ...Exception(Checked Exception)
* `java.lang.Exception` 클래스의 서브 클래스들 중에서 `RuntimeException`이 아닌 예외들
* 애플리케이션 코드에서 `try-catch`/`throws`로 처리해야 하는 예외들

## 2. 예외 처리할 때 하면 안되는 것
### 2.1. 예외를 잡아서 아무것도 하지 않기
```java
try {
    // 예외가 발생할 수 있는 코드
} catch (Exception e) {
    // 아무것도 하지 않음
}
```
* 예외가 발생했을 때 아무런 조치를 취하지 않으면, 문제의 원인을 파악하기 어렵고, 프로그램이 예상치 못한 동작을 할 수 있음
* 코딩 연습이나 예제 만드는 경우에라도 하지 말자.

### 2.2. 예외 잡아서 로그만 남기는 것
```java
try {
    // 예외가 발생할 수 있는 코드
} catch (Exception e) {
 e.printStackTrace(); // 로그만 남김
}
```
* 로그만 남기는건 예외를 처리한게 아니다
* 적절하게 복구하던지, 작업을 중단시키고 운영자나 개발자에게 분명하게 통보되어야한다.

### 2.3. 무책임하게 throws
```java
public void someMethod() throws Exception {
    // 예외가 발생할 수 있는 코드
}
```
* 메서드 시그니처에 `throws Exception`을 선언하는 것은 예외를 처리하지 않고 호출자에게 떠넘기는 것
* 호출자도 `throws Exception`을 선언해야 하고, 결국 예외가 처리되지 않고 프로그램 전체에 퍼질 수 있음

## 3. 예외 처리 방법
### 3.1. 예외 복구
* 예외상황을 파악하고 문제를 해결해서 정상 상태로 돌려놓는 것
* 방법:
  * 예외로 인해 기본 작업흐름 불가능 → 대체 작업흐름으로 복구
  * 재시도

### 3.2. 예외처리 회피
* throws 키워드를 사용해서 예외를 호출자에게 떠넘기는 것
* 주의: 의도가 분명해야함. 무책임하게 throws 하는 것은 안됨.

### 3.3. 예외 전환
* 예외를 다른 예외로 감싸서 던지는 것
* 예외의 원인을 숨기지 않고, 더 구체적이거나 의미있는 예외로 전환해서 던지는 것
* 기존 예외를 포함해주는 것이 좋다(중첩 예외)

## 4. RuntimeException의 보편화
* 자바 엔터프라이즈가 서버 환경으로 변하면서 수많은 사용자의 요청이 독립적인 작업으로 취급되고 
  특정 계층에서 예외가 발생했을 때 작업을 복구할 방법이 없어짐
* 그래서 요즘 CheckedException은 거의 사라지고, RuntimeException이 보편화됨

## 5. 애플리케이션 예외
* 시스템 또는 외부의 예외 상황이 원인이 아니라 애플리케이션 자체의 로직에 의해 의도적으로 발생시키고, 
  반드시 처리하도록 요구하는 예외를 말함
* 책에서는 체크 예외로 만들라고 하는데 개인적으로 ExceptionAdvice같은게 있는 상황에서 굳이 체크 예외로 만들 필요는 없다고 생각한다.

## 6. JDBC 예외
### 6.1. JDBC 배경 정보
* JDBC는 자바에서 데이터베이스에 접근하기 위한 API
* JDBC의 Connection, Statement, ResultSet 등의 클래스는 모두 `java.sql` 패키지에 포함되어 있음
* 다양한 DB가 JDBC API를 구현한 드라이버를 제공하기 때문에, JDBC를 사용하면 특정 DB에 종속되지 않고 데이터베이스 작업을 수행할 수 있음
* 하지만 각 DB마다 "비표준 SQL"이 존재하고, "다양한 예외"가 발생할 수 있어 JDBC는 "SQLException"이라는 체크 예외 단 하나만을 제공함.

### 6.2. SQLException
* `java.sql.SQLException`으로 Checked Exception 이다.
* JDBC 작업 중 발생할 수 있는 모든 예외가 SQLException으로 표현된다.

### 6.3. DataAccessException
* Spring의 JdbcTemplate이 JDBC 예외를 `org.springframework.dao.DataAccessException`으로 추상화하여 제공하는 것.
* DataAccessException은 SQLException을 포함하는 중첩 예외로, JDBC 예외의 원인을 숨기지 않고, 더 구체적이거나 의미있는 예외로 전환해서 던지는 역할을 한다.
* RuntimeException 이다.
* 각 DB의 예외를 적절한 DataAccessException 서브 클래스로 전환하지만, "일부"는 동일한 상황에서 다른 DataAccessException으로 전환될 수 있다.
* 따라서 DB가 변경되면 예외의 종류도 달라질 수 있어 
  DataAccessException을 직접 분기문에 이용한다면 어느정도 DB에 종속적일 수 있다는 걸 염두해야한다.
