# 5장. 서비스 추상화

## 1. [기능 추가] 파트에서의 인상깊던 부분
### 1.1. 데이터"상태"값 설정
* DB: "상태"에 대한 값을 varchar이 아닌 INT로 설정
  * → DB 용량을 많이 차지하지 않고 가볍기 때문(현재는 큰 차이 없다고 들었던 것 같다)
* App: "상태"에 대한 값을 Enum으로 설정 
  * → 코드에서 가독성이 좋고, 실수로 다른 값이 들어가는 것을 방지할 수 있기 때문
  * 단, DB가 INT로 설정되어있으면 값 변환에 대해 신경써야함. 그냥 DB도 ENUM 처리하는게 좋을 듯

### 1.2. TDD 방식대로 작성했다는 것
* 테스트 코드 작성 → 실제 코드 작성 → 리팩토링
* 특히 JDBC 사용할 때 오타는 컴파일 과정에서 검증이 되지 않으므로 테스트 코드가 더 큰 위력 발휘

### 1.3. 테스트 범위 설정
* 현재 테스트는 통합테스트로 진행(이후 변경 예정이라고 언급)
* 테스트 데이터를 "경계값"으로 설정하여 테스트의 신뢰도를 높임

### 1.4. 테스트 중복 부분을 "픽처스"로 제거
```java
@Test
public void upgradeLevels() {
    // 생략
    checked(user.get(0), Level.BASIC);
    checked(user.get(1), Level.SILVER);
}
private void checked(User user, Level expectedLevel) {
    assertThat(user.getLevel()).isEqualTo(expectedLevel);
}
```
## 2. [코드 개선]파트에서 인상깊던 부분
> 1. 코드에 중복된 부분이 없는가? 
> 2. 코드가 무엇을 하는 것인지 이해하기 불편하지 않는가? 
> 3. 코드가 자신이 있어야 할 자리에 있는가? 
> 4. 앞으로 변경이 일어난다면 어떤 것이 있을 수 있고, 그 변화에 쉽게 대응할 수 있게 작성되어있는가?

### 2.1. 분기문 개선
#### 1) 분기문을 메서드로 추출하여 [명확한 의미 부여 + 중복 제거]
```java
if (user.getLevel() == Level.BASIC && user.getLogin() >= 50) {
    user.setLevel(Level.SILVER);
    changed = true;
}
else if(user.getLevel() == Level.SILVER && user.getRecommend() >= 30) {
    user.setLevel(Level.GOLD);
    changed = true;
} else {
    changed = false;
}
```
```java
if (canUpgradeLevel(user)) {
    upgradeLevel(user);
}
```

#### 2) 분기문에서 else로 퉁치지 말고 현재 가능한 상황만을 명확히 표현
```java
private boolean canUpgradeLevel(User user) {
    if (user.getLevel() == Level.BASIC) return (user.getLogin() >= 50);
    if (user.getLevel() == Level.SILVER) return (user.getRecommend() >= 30);
    if (user.getLevel() == Level.GOLD) return false;
    throw new IllegalArgumentException("Unknown Level: " + user.getLevel());
}
```
* 현재 가능한 상황만을 명확히 표현하여, 새로운 레벨이 추가되었을 때 코드 수정이 반드시 필요함을 Exception으로 알림

### 2.2. 객체의 책임을 명확히 분리
#### 1) Enum에 변화를 Enum이 책임지도록 개선
```java
public enum Level {
    BASIC(1, null), SILVER(2, BASIC), GOLD(3, SILVER);
    private final int value;
    private final Level next;
    Level(int value, Level next) {
        this.value = value;
        this.next = next;
    }
    public int intValue() { return value; }
    public Level nextLevel() { return next; }
}
```
* Enum에 다음 레벨에 대한 정보를 추가하여, 새로운 레벨이 추가되었을 때 Enum만 수정하면 되도록 개선
* 서비스 레이어는 Enum의 nextLevel() 메서드만 호출하면 되므로, 서비스 레이어의 책임을 확실히 분리

#### 2) User 객체가 자신의 레벨을 스스로 업그레이드하도록 개선
```java
public void upgradeLevel() {
    Level nextLevel = level.nextLevel();
    if (nextLevel == null) {
        throw new IllegalStateException(this + "은 업그레이드가 불가능합니다.");
    } else {
        level = nextLevel;
    }
}
```
* User 객체가 자신의 레벨을 스스로 업그레이드하도록 개선하여, 서비스 레이어는 User 객체의 upgradeLevel() 메서드만 호출하면 되도록 개선

### 2.3. 앞으로 변경이 일어날 수 있는 부분을 추상화
#### 1) Policy 인터페이스를 만들어서, 레벨 업그레이드 정책을 추상화
```java
public interface UserLevelUpgradePolicy {
    boolean canUpgradeLevel(User user);
    void upgradeLevel(User user);
}
```
* 레벨 업그레이드 정책을 UserLevelUpgradePolicy 인터페이스로 추상화하여, 새로운 레벨 업그레이드 정책이 추가되었을 때 UserServiceImpl 클래스는 수정할 필요가 없도록 개선

## 3. Transaction 추상화
> 블로그에 작성한 내용 제외

### 3.1. 용어 정리
* 로컬 트랜잭션: 하나의 DB Connection 안에서 만들어지는 트랜잭션 → JDBC 트랜잭션
* 글로벌 트랜잭션: 여러 DB Connection 또는 다른 자원까지 포함하는 트랜잭션 → JTA(Java Transaction API) 트랜잭션

### 3.2. 트랜잭션 경계 설정
* 모든 트랜잭션은 시작하는 지점과 끝나는 지점이 존재
  * 시작하는 지점: autocommit = false
  * 끝나는 지점: commit() 또는 rollback()
* 트랜잭션 내부에서는 모두 동일한 Connection을 사용해야 함
  * 트랜잭션은 Connection 단위로 관리되기 때문

```java
Connection c = dataSource.getConnection();
c.setAutoCommit(false); // 트랜잭션 시작

try {
    // 비즈니스 로직 수행
    c.commit();
} catch (Exception e) {
    c.rollback();
    throw e;
} finally {
    c.close();
}
```

### 3.3. 트랜잭션 동기화
* 트랜잭션 동기화: 트랜잭션이 시작된 이후에 트랜잭션에 참여하는 모든 코드가 동일한 Connection을 사용하도록 보장하는 것
* TransactionSynchronizationManager: 트랜잭션 동기화를 지원하는 클래스
* JdbcTemplate은 내부에서 TransactionSynchronizationManager를 사용하여 트랜잭션 동기화를 지원
```java
 public void upgradeLevels() {
  TransactionSynchronizationManager.initSynchronization();
  Connection c = DataSourceUtils.getConnection(dataSource);
  c.setAutoCommit(false);
    try {
        // 비즈니스 로직 수행
        c.commit();
    } catch (Exception e) {
        c.rollback();
        throw e;
    } finally {
        DataSourceUtils.releaseConnection(c, dataSource);
        TransactionSynchronizationManager.unbindResource(dataSource);
        TransactionSynchronizationManager.clearSynchronization();
    }
}
```
* `DataSourceUtils.getConnection()` 메서드는 트랜잭션이 시작된 이후에 호출되면, 트랜잭션에 참여하는 모든 코드가 동일한 Connection을 사용하도록 보장
  * `dataSource.getConnection()` 메서드는 매번 새로운 Connection 반환
* 문제:
  * "로컬 트랜잭션"이 아니라 "글로벌 트랜잭션"이라면, 동일한 Connection 사용해도 트랜잭션이 보장되지 않음
  * 즉, 환경이 변하면 비즈니스 로직이 변하지 않았음에도 코드 수정이 필요함
* JTA 참고
  ```java
  public void upgradeLevels() {
    InitialContext initialContext = new InitialContext();
    UserTransaction tx = (UserTransaction) ctx.lookup("java:comp/UserTransaction");
  
    tx.begin();
    Connection c = dataSource.getConnection();
    try {
        // 비즈니스 로직 수행
        tx.commit();
    } catch (Exception e) {
        tx.rollback();
        throw e;
    } finally {
        c.close();
    }
  }
  ```

### 3.4. 트랜잭션 추상화
* 트랜잭션을 처리하는 코드는 일정한 패턴을 가지고 있음
  * 트랜잭션 시작 → 비즈니스 로직 수행 → 트랜잭션 종료
* 트랜잭션 추상화: 트랜잭션을 처리하는 코드를 일정한 패턴으로 추상화하여, 트랜잭션 처리 코드를 재사용할 수 있도록 하는 것
* Spring에서는 `PlatformTransactionManager` 인터페이스를 통해 트랜잭션 추상화를 지원
  ```java
  public void upgradeLevels() {
    PlatformTransactionManager txManager = new DataSourceTransactionManager(dataSource);
    TransactionStatus status = txManager.getTransaction(new DefaultTransactionDefinition());
    try {
        // 비즈니스 로직 수행
        txManager.commit(status);
    } catch (Exception e) {
        txManager.rollback(status);
        throw e;
    }
  }
  ```
